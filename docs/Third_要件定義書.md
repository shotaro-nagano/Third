# Third（仮）要件定義書

> 行きつけの店にいる間だけ、同じ店に通い続けた常連同士が、ゆるい合図で知り合えるSNS。
> 店を出れば関係はリセットされる。追ってこない、盛らない、気楽。

- **ドキュメント種別**: 要件定義書（MVP / v0.1）
- **対象プラットフォーム**: スマートフォン優先。Web・iOS・Windows すべてに対応
- **ステータス**: 壁打ち確定事項を反映した初版。未確定箇所は「未確定」と明記

## 0. このドキュメントの読み方

本書は「決まったこと」と「これから決めること」を区別して書いている。各項目の末尾に確度を示す。

- **確定**: 壁打ちで合意済み。設計の前提とする
- **推奨**: 設計上の推奨。着手前に最終判断が必要
- **未確定**: 今後の検証・判断が必要

数値（市場規模・利用率など）は外部の一次情報での裏取りを着手前の第0タスクとする。

## 1. 背景と課題

### 1.1 解きたい課題

都市生活者にとっての「サードプレイス」（家でも職場でもない第三の居場所）が失われつつある。かつて行きつけの店には、深く話すわけではないが顔見知りで、店に行けば誰かがいる、というゆるい常連関係があった。この「気楽な常連関係」は、既存のSNS（X・Instagram・TikTok）でもグループチャット（LINE・Discord）でも再現できていない。

既存SNSの構造的な問題:

- 投稿が「観客」に向いており、盛る・映える圧力がある
- 関係がつながりっぱなしで、距離を置きづらい
- 生成AIによりコンテンツと偽アカウントが無限に増え、「本物の人間・実際に会うこと」が希少になっている

### 1.2 なぜ今か

- 孤独・サードプレイス喪失が都市生活者の本物の痛みとして広がっている
- 生成AI時代に「実在する人間が、実際に同じ場所にいた」という事実の価値が相対的に上がっている
- 巨大SNSは広告＝無限フィード依存のため、「閉じた・その場限り・少人数」の方向へ全振りできない（＝構造的に追随しづらい）

### 1.3 コンセプト一文（確定）

> 行きつけの立ち飲み屋にいる間だけ、同じ店に通い続けた常連同士が、ゆるい合図で知り合えるSNS。店を出れば関係はリセットされる。

### 2.1 設計の芯（一貫性の確認）

場所主軸・常連限定・その場限り・片方向の合図——すべての要素が「**気軽だけど安全な常連の場**」という一点に向かって整列している。これは「つながりっぱなしのSNS」への構造的なアンチテーゼである。新機能を検討する際は、この芯に奉仕するかどうかを判断基準とする。

## 3. ターゲット

### 3.1 最初の1人（プライマリ・ペルソナ）— 未確定（要詰め）

立ち上げは「創業者自身がヘビーユーザーである一店舗」から始めるため、最初のペルソナは創業者の行きつけの常連像に合わせて確定する。仮の像:

- 20代後半〜40代。都市部で働き、週に複数回ワイワイ系の立ち飲み屋に通う
- 一人でふらっと飲みに行くことが多く、その場の偶然の会話を楽しみにしている
- 「常連同士もっと知り合えたらいいのに、きっかけがない」と感じている

### 3.2 広げる順番（確定）

1. 最初の一店舗（創業者の行きつけ）を満たし切る
2. 同エリアの近隣の立ち飲み屋へ横展開
3. 他エリア／他都市の立ち飲み文化へ
4. （将来）立ち飲み以外の店種・場所種へ

> 原則: 「すべての場所」を同時に狙わない。一点を制圧してから広げる。

### 3.3 明確に守るべきユーザー

一人で来た女性が安心して使えること。性別比が崩れた場（男性ばかり）になった瞬間に女性が離脱し、場が死ぬ。安全設計（§7）はこの観点を最優先で設計する。

## 4. コア体験（ユーザーフロー）

店に入ってから出るまでの一連の流れ。これがThirdの中核。

1. **チェックイン** — 店に着いてアプリを開く。位置情報で「今この店にいる」が記録される。
2. **常連が見える** — 今その店にいる常連がゆるく一覧表示される。「こっそりチェックイン」を選べば、自分はリストに載らず、見るだけにできる。
3. **ゆるい合図** — 気になった常連に「乾杯しませんか？」のような軽い合図を送れる。片方向。一晩に送れるのは1〜2回まで。
4. **現実で合流** — 同じ店の中にいるので、あとは目を合わせて隣へ。会話そのものは現実に明け渡す。
5. **店を出たら切れる** — その場限り。やりとりは残らない。また同じ店で会えば再開する。

### 4.1 体験の境界（やること / やらないこと）

**やること**

- 「今この店に、どんな常連がいるか」を見せる
- 知り合う「きっかけのひと押し」を提供する（合図）
- 通い続けた履歴を「常連」という信頼に変える

**やらないこと（MVP）**

- 店外・営業時間外の個別メッセージ機能（その場限りの思想に反する）
- フォロワー数・いいね数・公開プロフィールなどの「映え」を促す指標
- 連絡先交換の強制や仲介（発展は現実とユーザーに委ねる）

## 6. 非機能要件

### 6.1 プラットフォーム対応（確定）

スマホ優先で設計し、Web・iOS・Windows すべてに対応する。これを現実的なコストで満たすため、**単一コードベースのクロスプラットフォーム構成**を推奨する。

| 対象 | 提供形態 | 推奨実装 |
| --- | --- | --- |
| iOS | ネイティブアプリ（App Store） | Flutter または React Native |
| Android | ネイティブアプリ（Google Play） | 同上（同一コードベース） |
| Web | レスポンシブWebアプリ（PWA） | Flutter Web または React |
| Windows | デスクトップアプリ | PWAインストール または Flutter Desktop |

> 補足: コア体験が「店内でのチェックイン」である以上、主たる利用はスマホになる。Web/Windowsは閲覧・設定・店側管理など補助的役割が中心。位置情報が要となるため、デスクトップでのチェックインは制限する想定。

### 6.2 品質の床（frontend-design の原則を全プラットフォームで担保）

- レスポンシブ: スマホ最小幅まで破綻しない
- アクセシビリティ: キーボードフォーカスが見える、`prefers-reduced-motion` を尊重、コントラスト確保
- パフォーマンス: チェックインから常連リスト表示まで体感1秒以内を目標
- ダークモード: 店内＝夜の利用が主のため、ダークを第一級として設計

### 6.3 プライバシー・データ保護

- 位置情報は「チェックイン判定」に必要な最小限のみ取得・保持
- やりとり（合図）は店を出たら破棄する（その場限りの思想をデータ設計でも担保）
- 各国の個人情報保護法（日本の個人情報保護法、EUのGDPR等）への準拠を世界展開前に確認

### 6.4 セキュリティ

- 通信のTLS暗号化、認証情報の安全な保管
- 位置偽装・多重アカウントへの抑止（§7.3）

## 7. 信頼・安全設計（最重要）

「実際に知り合う」「夜・飲酒・初対面」という最もリスクの高い組み合わせを、マスター常駐なしで運用する。安全設計はThirdの生命線。

### 7.1 三層の安全装置（確定）

1. **通い続けたコスト＝信頼**: 別々の日に5回・各回一定滞在しないと常連になれない。出会い目的の人間が「ナンパのために週数回・数週間、同じ店に通い続ける」コストを払えない。通うコスト自体が悪意のフィルターになる。
2. **その場限り**: つながれるのは「今その店にいる常連」同士のみ。店を出ればやりとりは切れる。店外の付きまとい・DMナンパが構造的に不可能。
3. **片方向の合図＋受け手の防御**: 送る側の気軽さは残しつつ、安全の重心を「上限」と「受け手の守り」に置く。

### 7.2 片方向合図の必須セット（確定）

片方向を選んだ代償を払うため、以下3点は譲らず実装する。

1. **上限を強く絞る**: 一晩に送れる合図は1〜2回。乱射を不可能にする（特定の一人への集中砲火を防ぐ最大のブレーキ）。
2. **受け手に拒否の退路**: 合図は軽く届くだけ。無視しても角が立たない（送り手に伝わらない）。「そっとブロック」をワンタップで。受け手が常に主導権を持つ。
3. **可視性は安全側がデフォルト**: 「こっそりチェックイン」を必ず備える。特に一人で来た人が、見られずに様子を見られる退路。

### 7.3 位置偽装への方針（推奨）

完全防御は不可能。「割に合わなくする」方向で抑止する。別日×5回×滞在時間を稼ぐ手間に対し、得られるのは「一立ち飲み屋の常連という資格」のみ。偽装コストが見返りを上回るため構造的に抑止される。残余リスクは通報（F-2）で補完する。

### 7.4 性別比の偏りへの対処（P1）

男性に偏った場は女性離脱を招き場が死ぬ。検知と対処の仕組みを設ける（初期は運営の手動運用、将来は自動化）。具体的な介入方法は未確定。

## 8. 収益（後回し・無料導入）

導入は無料（確定）。人が集まるまで価値が出ないため、まず無料で集め切ってから回収する。

将来の収益候補（未確定・優先度低）:

- 公認店（マスター）向けの有料機能（告知・常連分析など）
- ユーザー向けプレミアム（追加の合図枠、店をまたいだ常連実績など）
- 店との送客・イベント連携

> 注意: 収益化が「映え」や「つながりっぱなし」を促す方向に働くと設計の芯を壊す。収益機能は芯への適合を必ず確認する。

## 9. 立ち上げ戦略（初日の過疎対策・確定）

SNS最大の急所「誰もいない部屋」を、現実の力で越える。

1. **一店舗に集中投入**: アプリで広く薄く配らず、創業者の行きつけ一店に薪を集めて火をつける。
2. **三段の初期ユーザー獲得**:
   - 行きつけの店でリアルに声をかけて集める
   - マスターに後押ししてもらう（容認モデル）
   - 自分の人脈・SNSで「○○（店名）で始めた」と告知し外からも常連を呼ぶ
3. **満たし切ってから次へ**: 一店舗が「開けば誰か常連がいる」状態になってから、同エリアの近隣店へ広げる。

### 9.1 コンシェルジュ型MVP（着手前の検証・強く推奨）

コードを書く前に、思想だけを現実でテストする。

- 行きつけの店で、常連10〜20人のグループ（LINE等）を作る
- 4週間、「店にいる時だけ」「軽い合図だけ」のルールを手動運用する
- 測る指標: ①常連同士が実際に知り合えたか ②使い続けたか ③「またやりたい」と思ったか
- ここで仮説が死ぬなら、開発に進まない判断ができる

## 10. 成功指標（KPI）

### 10.1 北極星指標（案・未確定）

> 「行きつけの一店で、別の常連と新しく知り合えた人の割合」

Thirdの存在価値そのもの。これが立たなければ機能をいくら足しても意味がない。

### 10.2 補助指標

- 一店舗あたりの「常連」人数（密度。過疎の反対）
- チェックイン後に常連リストを開く率（コア体験への到達）
- 合図の送信→現実合流の発生率
- 週次の再訪率（その場限りでも「また来る」が起きているか）
- 安全指標: 通報率、そっとブロック率、性別比

## 11. 技術構成（推奨・MVP）

| レイヤ | 推奨 | 備考 |
| --- | --- | --- |
| クライアント | Flutter（iOS / Android / Web / Windows を単一コードベース） | スマホ優先・全プラットフォーム対応の要件に最適 |

| 通知 | プッシュ通知（FCM / APNs） | 合図の受信通知 |

> MVPは「最初の一店舗」が回ればよいので、自前構成を最小化し、検証速度を優先する。

## 付録A. 用語

- **部屋**: アプリ上で一つの店舗を指す単位
- **常連**: 別々の日に5回・各回一定滞在した、その店の認定ユーザー
- **合図**: 今いる常連へ送る、片方向の軽いリアクション
